# 基本共同性原則（Shared Engineering & Governance Principles）

本文件整理所有「中層、可重複使用、跨檔案一致」的工程與治理原則。

---

## 一、進度治理（強制）

- 子進度只能用：**0 / 50 / 80 / 100%**
- 每次變更後必須回報：
  1) 本次對應的子進度名稱  
  2) 子進度：修改前% → 修改後%  
  3) 總體進度：修改前% → 修改後%  
  4) 本次變更內容（檔案與外顯行為）  
  5) 驗收狀態（建置／測試／驗證）  
  6) 下一步（只允許一個明確行動）

---

## 二、指令包交付規則（給 Codex / Cursor）（強制）

只要牽涉代碼或檔案修改，我輸出必須符合：

- **單一整塊**、一次貼上即可用
- 指令包必含：
  - 目標（要達到的外顯行為）
  - 變更範圍（哪些檔案／哪些模組）
  - 具體修改步驟（可直接照做）
  - 驗證步驟（怎麼確定沒壞）
  - 回滾方式（出問題怎麼撤）

---

## 三、你端驗收回報格式（白話、最少負擔）

你只需要回報三件事：

1. 你操作了什麼（例如：點了哪個按鈕、走了哪個流程）
2. 你看到什麼差別（前 vs 後）
3. 有沒有符合你要的（是／否）與卡在哪

---

## 四、資料結構一致性原則

- 題目與選項的最小單位：**選項的中繼資料（choice meta）**
- 新舊題庫必須可轉換到同一語意格式
- 欄位命名不得漂浮：同一概念只能有一種寫法

---

## 五、可解釋性原則

- 每一個分數必須能回溯來源（題目、選項、權重、規則）
- 每一個母題／標籤必須有明確映射規則
- AI 文字輸出不得反向影響 scoring

---

## 六、UX 與語言原則

- 去病理化：不用診斷語言
- 先身體、後情緒：以體感與生活語言承載訊號
- 使用者能理解，但說不出自己「被測什麼」

---

## 七、錯誤處理與技術債

- 只有 console 訊息不算錯誤處理
- 使用者端必須看到可理解的 fallback
- 未解決事項必須進入 Pending Issues / Backlog（不可口頭記憶）
